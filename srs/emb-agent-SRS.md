# 《嵌入式系统程序优化AGENT》软件需求规格说明书

## 1 引言
*   **1.1 编写目的**
    *   为AGENT的开发提供统一、明确的需求基础；为后续AGENT的设计、实现、测试和评估提供依据；明确AGENT的功能边界、性能指标和评估方法。
*   **1.2 背景**
    *   **项目名称**：嵌入式系统程序开发AGENT。
    *   **项目定位**：一个智能化的辅助开发工具，旨在通过大语言模型（LLM）与代码工程化能力的结合，自动或半自动地完成嵌入式项目从需求分析到原型验证的全流程。
    *   **项目目标**：
        1.  开发出一个可交互的AGENT程序。
        2.  利用该AGENT，为飞腾开发板生成一个具体的嵌入式应用项目方案。
        3.  根据AGENT生成的方案，在飞腾开发板上搭建原型，并评估方案的可行性、完整度和AGENT的性能。
        4.  基于评估结果，提出对AGENT的改进设想。
    *   **主要用户痛点**：嵌入式开发门槛高、项目搭建繁琐、跨平台适配（如国产飞腾平台）经验不足、技术方案调研耗时。
    *   **目标用户**：嵌入式系统开发者、学生、研究人员、对国产化平台感兴趣的工程师。

*   **1.3 参考资料**
    *   LLM相关技术文档
    *   飞腾开发板硬件手册、软件开发指南。
    *   嵌入式Linux开发相关标准与文档

*   **1.4 假定与约束**
    *   **业务约束**：AGENT生成的代码或方案需要经过人工审核，不能直接部署到生产环境。LLM返回的内容可能存在幻觉，需要后续验证。
    *   **团队约束**：开发周期约为1-2月。开发语言为Python，依赖LLM API。硬件环境为飞腾开发板、串口线、网络等。


## 2 功能需求
*   **2.1 系统范围**
    *   定义AGENT的核心功能模块：
        1.  **用户交互与需求解析模块**：接收用户自然语言描述的项目需求。
        2.  **知识库与模板管理模块**：管理嵌入式开发的知识（如飞腾平台特性、驱动开发流程）和项目模板。
        3.  **可行性评估模块**：模拟或基于经验分析生成方案的可行性，列出潜在风险。
        4.  **方案生成与代码编写模块**：根据需求生成项目架构、代码、构建脚本。
        5.  **原型搭建与测试模块**：提供自动在飞腾开发板上部署、运行和测试原型的功能。
        6.  **问题排查与自动修复模块**：根据用户反馈的错误信息，分析问题并提供修复建议。

*   **2.2 系统总体流程**
    *   **文本描述流程**：
        1.  用户向AGENT描述项目需求（例如：“在飞腾开发板上创建一个LED闪烁的驱动和应用”）。
        2.  AGENT解析需求，拆解任务。
        3.  AGENT生成项目方案（包括软硬件选型、架构设计、代码模块划分）。
        4.  AGENT生成具体的代码文件和编译脚本。
        5.  自动将生成的代码在飞腾开发板上进行部署和测试。
        6.  用户反馈测试结果或遇到的问题。
        7.  AGENT根据反馈协助调试或优化方案。

*   **2.3 需求分析建模**

#### **UC-01 解析用户需求（Parse User Requirement）**

**参与者：**  
用户（通过命令行或Web界面）

**前置条件：**  
AGENT已启动，与LLM服务连接正常。

**后置条件：**  
系统生成结构化的需求描述（功能列表、非功能约束、目标平台）。

**基本流程：**
1. 用户输入自然语言描述的项目需求。
2. AGENT调用LLM将需求解析为结构化格式，提取关键要素。
3. AGENT向用户展示解析结果，并请求确认。
4. 用户确认无误后，需求解析完成。

**异常流程：**
- 2a. LLM API超时或返回格式错误 → AGENT重试最多3次，若仍失败则提示“需求解析失败，请稍后重试”。
- 3a. 用户认为解析结果不准确 → 用户可以手动编辑结构化需求，AGENT接受修改并更新。

---

#### **UC-02 检索知识库（Retrieve Knowledge）**

**参与者：**  
AGENT内部（由其他用例触发）

**前置条件：**  
知识库已初始化，包含飞腾开发板手册、Linux驱动开发指南等资料。

**后置条件：**  
返回与当前需求最相关的前K条知识片段。

**基本流程：**
1. 系统接收到一个查询（如“飞腾GPIO驱动开发”）。
2. 将查询转换为向量，在知识库中检索相似度最高的条目。
3. 返回匹配的知识片段列表。
4. 调用LLM对这些片段进行提炼，生成可直接使用的摘要或代码模板。

**异常流程：**
- 2a. 知识库服务不可用 → 记录错误，返回空列表，后续依赖LLM原生知识。
- 3a. 检索结果为空 → 返回空列表，LLM仅使用自身知识。

---

#### **UC-03 生成项目方案（Generate Project Plan）**

**参与者：**  
用户（可选确认），AGENT内部

**前置条件：**  
已完成需求解析（UC-01）。

**后置条件：**  
生成一份包含项目架构、模块划分、开发步骤的方案文档。

**基本流程：**
1. AGENT根据结构化需求，调用LLM生成项目方案初稿。
2. 若需要，AGENT通过UC-02检索相关开发模板或最佳实践，融入方案。
3. 系统将方案以文本或图表形式展示给用户。
4. 用户可对方案提出修改意见，AGENT根据反馈迭代优化。
5. 用户确认方案后，流程结束。

**异常流程：**
- 1a. LLM生成内容为空或格式异常 → 重试或回退到备用提示词。
- 4a. 用户连续多次修改仍不满意 → AGENT记录失败案例，用于后续改进。

---

#### **UC-04 生成代码文件（Generate Code Files）**

**参与者：**  
AGENT内部

**前置条件：**  
已确认项目方案（UC-03）。

**后置条件：**  
在指定目录生成项目所需的所有源代码文件、头文件、Makefile/CMakeLists.txt等。

**基本流程：**
1. AGENT根据方案中列出的模块，逐一调用LLM生成对应代码文件。
2. 对于需要平台特定API的部分，AGENT通过UC-02检索飞腾开发板官方示例或手册，确保API使用正确。
3. 生成完整的构建脚本。
4. 将生成的代码写入本地文件系统，并向用户展示文件列表。

**异常流程：**
- 1a. 某个模块代码生成失败 → 记录错误，跳过该模块，提示用户手动补充。
- 4a. 文件写入失败→ 提示用户检查环境。

---

#### **UC-05 读取代码文件（Read Code File）**

**参与者：**  
AGENT内部（用于代码审查、修改、测试分析等）

**前置条件：**  
存在可读的代码文件路径。

**后置条件：**  
AGENT获取文件内容，并可在后续任务中使用。

**基本流程：**
1. AGENT接收到一个文件路径。
2. 系统读取文件内容。
3. 将内容传递给LLM，用于分析或修改。
4. （可选）LLM返回文件内容的总结或关键信息。

**异常流程：**
- 2a. 文件不存在或无法读取 → 返回错误，终止当前任务。
- 2b. 文件格式非文本→ 提示“无法读取非文本文件”。

---

#### **UC-06 修改代码文件（Modify Code File）**

**参与者：**  
用户（或AGENT自动根据错误修复）

**前置条件：**  
已读取目标文件内容（UC-05）。

**后置条件：**  
文件内容被更新，修改后的文件保存到本地。

**基本流程：**
1. 用户（或自动流程）向AGENT提供修改指令。
2. AGENT调用LLM分析当前代码，生成修改后的代码块。
3. AGENT将修改后的代码写回原文件。
4. 向用户展示修改差异。

**异常流程：**
- 2a. LLM生成的修改不符合语法或逻辑 → AGENT再次尝试或回退，并提示用户手动干预。
- 3a. 写入文件失败 → 提示错误，保留原文件。

---

#### **UC-07 自动部署代码（Deploy to Target）**

**参与者：**  
AGENT内部

**前置条件：**  
已生成项目代码（UC-04），且飞腾开发板已通过网络（SSH/SCP）或串口连接。

**后置条件：**  
代码文件成功传输到开发板指定目录。

**基本流程：**
1. AGENT获取开发板连接参数（IP、用户名、密码/密钥）。
2. 使用SCP或rsync将本地代码目录同步到开发板目标路径。
3. 返回部署成功状态。

**异常流程：**
- 2a. 网络不可达或认证失败 → 提示用户检查连接配置，并中止部署。
- 2b. 目标路径无写入权限 → 提示用户修改权限或选择其他路径。

---

#### **UC-08 执行测试命令（Execute Test Commands）**

**参与者：**  
AGENT内部

**前置条件：**  
代码已部署到开发板（UC-07）。

**后置条件：**  
执行编译、运行等命令，并获取输出结果。

**基本流程：**
1. AGENT生成测试命令序列。
2. 通过SSH执行命令，实时捕获标准输出和标准错误。
3. 返回命令执行结果（成功/失败，输出日志）。

**异常流程：**
- 2a. SSH连接中断 → 重试3次，失败则提示“无法连接到开发板”。
- 2b. 命令执行超时（如超过30秒）→ 终止进程，返回超时错误。

---

#### **UC-09 分析错误日志（Analyze Error Log）**

**参与者：**  
AGENT内部

**前置条件：**  
从UC-08获取到包含错误信息的输出。

**后置条件：**  
AGENT定位错误原因，并可能触发UC-06或UC-10。

**基本流程：**
1. AGENT将错误日志传递给LLM，并附上相关代码片段。
2. LLM分析错误类型和具体位置。
3. AGENT向用户展示分析结果。

**异常流程：**
- 2a. LLM无法识别错误 → 返回“无法自动分析，请手动检查日志”。
- 2b. 错误日志为空 → 跳过分析。

---

#### **UC-10 提供修复建议（Provide Fix Suggestion）**

**参与者：**  
AGENT内部（可自动触发或用户请求）

**前置条件：**  
已完成错误分析（UC-09）。

**后置条件：**  
生成修复建议或直接修改代码。

**基本流程：**
1. AGENT根据UC-09的分析结果，调用LLM生成修复建议。
2. 若用户开启了自动修复模式，AGENT直接调用UC-06修改代码文件。
3. 若为手动模式，展示建议供用户参考。

**异常流程：**
- 1a. LLM生成的建议不切实际 → AGENT标记该错误为“待人工处理”。

---

#### **UC-11 生成评估报告（Generate Evaluation Report）**

**参与者：**  
用户、AGENT

**前置条件：**  
至少完成一次从需求到部署的完整流程。

**后置条件：**  
生成一份总结文档，包含任务完成情况、AGENT性能指标（响应时间、成功率、代码质量）、改进建议。

**基本流程：**
1. AGENT收集本次交互过程中的所有关键数据：需求复杂度、生成代码行数、LLM调用次数、错误次数、自动修复次数等。
2. 调用LLM对数据进行总结，生成评估报告。
3. 报告以Markdown或文本形式呈现给用户。
4. 用户可将报告导出或作为后续改进的输入。

**异常流程：**
- 2a. LLM生成报告失败 → 返回“评估报告生成失败”。

## 3 非功能需求
*   **3.1 性能要求**
    *   **响应时间**：对于简单的需求（如生成一个GPIO操作示例），AGENT整体响应时间（从输入到输出最终代码）应小于1min。对于复杂项目，应提供进度提示。
    *   **准确性**：生成的代码应满足语法正确性。对于飞腾平台相关的系统调用、库函数使用，应保证与平台手册一致。
    *   **方案可行性**：生成的方案在飞腾开发板上应能成功编译和运行。首次生成的方案可行性（无需人工修改代码即可运行）应不低于70%。

*   **3.2 数据管理能力**
    *   **知识库**：需要维护一个关于嵌入式开发、飞腾平台、常见驱动开发模式的向量知识库，以便LLM进行检索增强生成（RAG）。
    *   **会话管理**：能够保存用户与AGENT的多次对话历史，支持上下文关联的提问。

*   **3.3 安全要求**
    *   **API Key管理**：LLM API密钥必须安全存储，不能硬编码在代码中或泄露。
    *   **代码安全**：生成的代码不应包含明显的安全漏洞（如缓冲区溢出），应进行基础的代码静态检查。
    *   **操作安全**：提供的部署指引不应包含可能导致开发板损坏的指令（如错误的电压设置）。

*   **3.4 灵活性**
    *   **平台可扩展**：知识库应支持轻松添加新的开发板（如树莓派、其他ARM/国产芯片）和新的开发场景（如AI推理、网络应用）。
    *   **LLM可替换**：应能通过修改配置，切换不同的LLM后端（如DeepSeek, Qwen, ChatGPT等）。
    *   **评估指标可配置**：对AGENT性能的评估指标应允许用户自定义，如关注代码简洁性、性能、功耗等。

*   **3.5 其他专门要求**
    *   **可维护性**：AGENT自身的代码应有清晰的模块划分和注释，便于后续改进。
    *   **可评估性**：AGENT需要能够记录其内部的思考过程（Chain of Thought）和与LLM的交互日志，为性能评估提供数据支持。
    *   **用户友好**：交互界面可以是命令行或简单的Web UI，清晰展示AGENT的思考过程和输出结果。
